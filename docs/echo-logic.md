# EchoFit 回声逻辑说明

## 1. 文档目标

这份文档用于解释 EchoFit 当前“回声”功能的实际实现逻辑。

它不是产品理想设计稿，而是基于当前代码的真实行为总结，方便后续：

- 理解回声为什么会出现某条文案
- 判断现有算法是否满足产品预期
- 后续迭代时明确该改哪一层

当前实现的核心入口在：

- `backend/src/main/java/com/example/trainingecho/echo/EchoService.java`

## 2. 回声在什么时候生成

当前回声不是“随时动态实时重算”的，而是：

- 用户新建训练记录时立即生成一次
- 生成结果落库
- 后续查看记录时读取当时生成的结果

对应链路：

1. 前端调用 `POST /api/training-records`
2. 后端创建训练记录
3. `TrainingRecordService.createRecord(...)` 调用 `echoService.generateForNewRecord(...)`
4. 回声结果写入 `echo_history`
5. 前端后续通过 `GET /api/echo/by-record/{recordId}` 读取这条记录对应的回声

重要结论：

- 编辑训练记录不会重新生成回声
- 保存未来话不会立刻改写当前记录的回声
- 回声是“记录创建时的快照”

## 3. 当前回声来源分类

当前回声只有 3 种来源：

- `future_message`
- `fact`
- `system`

优先级固定如下：

1. 优先使用未来话
2. 没有未来话则尝试生成事实回声
3. 两者都没有时使用系统默认文案

也就是说，当前算法不是打分模型，而是一个固定优先级的规则引擎。

## 4. 整体算法流程

可以把当前逻辑理解成下面这段伪代码：

```text
创建训练记录时：
  先尝试挑一条可用的 future_message
  如果挑到了：
    生成“来自过去的你：...”类型回声
    更新该 future_message 的 used_count 和 last_used_at
    写入 echo_history
    结束

  如果没挑到 future_message：
    尝试根据最近 7 天训练记录生成一条 fact 文案
    如果能生成：
      写入 echo_history
      结束

  如果 fact 也生成不了：
    从默认系统文案里挑一条
    写入 echo_history
```

## 5. 未来话回声逻辑

### 5.1 什么是未来话

未来话由用户主动写入：

- 接口：`POST /api/echo/future-message`

请求体：

```json
{
  "recordId": "1912345678901234561",
  "content": "状态一般的时候，更要把节奏保住。"
}
```

### 5.2 未来话的保存规则

当前限制：

- 同一条训练记录最多只能保存 1 条未来话
- 内容不能为空
- 内容最大长度 50 字

保存后会发生两件事：

- 写入 `echo_message`
- 把这条未来话的 `id` 回写到训练记录的 `futureMessageId`

这主要用于：

- 记录这条训练记录关联了哪条未来话
- 在训练记录详情里展示 `futureMessagePreview`

### 5.3 新记录创建时，如何挑未来话

算法位置：

- `EchoService.pickFutureMessage(...)`

筛选条件：

- 必须属于当前用户
- `source = future_message`
- `relatedRecordId != 当前记录 ID`

也就是说：

- 不会把“今天刚写给自己的未来话”立刻回给今天这条记录
- 未来话的设计是给“之后的某次训练”使用

### 5.4 未来话的去重与排序规则

当前逻辑会先取出最近 3 条已经被回声使用过的未来话 ID：

- 来源表：`echo_history`
- 条件：`source = future_message`
- 排序：按时间倒序
- 只取最近 3 条

然后在所有候选未来话中，按下面顺序选：

1. `lastUsedAt` 越早越优先
2. 若 `lastUsedAt` 相同，再按 `id` 升序
3. 跳过最近 3 次已经用过的未来话
4. 返回第一条符合条件的消息

这意味着当前未来话策略更接近：

- “尽量不要重复用最近刚用过的未来话”
- “优先复用长期没用过的未来话”

而不是：

- 随机抽取
- 智能语义匹配
- 基于训练部位匹配

### 5.5 未来话被使用后会发生什么

一旦某条未来话被选中：

- `usedCount + 1`
- `lastUsedAt = 当前时间`
- 生成文案：

```text
来自过去的你：<用户写的未来话内容>
```

然后写入 `echo_history`。

## 6. 事实回声逻辑

如果没有可用未来话，系统会尝试生成“事实回声”。

算法位置：

- `EchoService.buildFactMessage(...)`

### 6.1 统计范围

当前事实回声只看：

- 当前用户
- 当前记录日期往前数 7 天
- 含当天

也就是说，如果今天是 `2026-05-17`，统计范围是：

- `2026-05-11` 到 `2026-05-17`

并且会包含今天刚刚新建的这条记录。

### 6.2 规则一：近 7 天首次训练某个部位

系统会先读取当前记录的训练部位列表，例如：

- `["legs", "core"]`

然后按当前记录里的部位顺序逐个检查：

- 统计这个部位在最近 7 天所有记录里一共出现了几次
- 如果某个部位计数正好等于 `1`
- 说明它只在今天这条记录里出现过
- 则生成：

```text
今天是你近 7 天第一次腿训练。
```

这里有几个细节：

- 文案优先级高于“第 N 次训练”
- 如果当前记录里有多个部位都满足“近 7 天第一次”，只取第一个命中的部位
- 中文部位名称来自 `TrainingPart.labelOf(...)`

当前映射是：

- `chest` -> 胸
- `back` -> 背
- `shoulder` -> 肩
- `legs` -> 腿
- `arms` -> 手臂
- `core` -> 核心
- `cardio` -> 有氧
- `stretch` -> 拉伸

### 6.3 规则二：近 7 天第 N 次训练

如果没有命中“首次部位训练”，则继续判断最近 7 天训练天数。

当前实现直接取最近 7 天记录数：

```text
这是你近 7 天里的第 3 次训练。
```

这里的计数包含今天这条新记录。

因为当前业务规则里：

- 同一用户同一天只能创建 1 条训练记录

所以“记录数”也等价于“训练天数”。

### 6.4 什么时候不会生成事实回声

理论上，只要刚创建了训练记录，最近 7 天至少会有当前这一条记录，所以基本都会至少命中“第 1 次训练”这条逻辑。

因此当前代码里：

- `fact` 分支实际上很容易命中
- `system` 分支属于更靠后的兜底逻辑

## 7. 系统默认回声逻辑

当未来话和事实回声都不可用时，系统才会退回默认文案。

当前默认文案只有 3 条：

1. `节奏不会一下子成形，但会慢慢留下痕迹。`
2. `不必每次都状态最好，回来就很好。`
3. `在普通的一天完成训练，也很难得。`

算法位置：

- `EchoService.pickDefaultMessage(...)`

### 7.1 当前默认文案的选择策略

系统会查当前用户最近一条 `source = system` 的历史回声。

然后：

- 如果从没生成过系统文案，直接取第 1 条
- 如果生成过，就从默认列表里找一条和上一条不相同的文案
- 找到第一条不同的就返回

这意味着当前行为是：

- 只避免和“上一条系统文案”完全重复
- 不是完整轮询
- 也不是随机

例如：

- 上一条系统文案是第 1 条
- 那下一次会直接返回第 2 条

## 8. 回声结果是如何落库的

当前有两张和回声直接相关的表：

- `echo_message`
- `echo_history`

### 8.1 `echo_message`

主要存：

- 用户写的未来话
- 未来话来源记录 `relatedRecordId`
- 使用次数 `usedCount`
- 上次使用时间 `lastUsedAt`

它更像“可复用消息池”。

### 8.2 `echo_history`

主要存每次真正展示给某条训练记录的回声结果：

- `user_id`
- `record_id`
- `echo_message_id`
- `source`
- `content`
- `created_at`

它更像“最终生成结果快照”。

### 8.3 为什么要分两张表

因为当前设计里：

- 用户写的未来话是“候选素材”
- 某次训练最终实际收到哪条回声，是“生成结果”

两者不是同一个概念，所以分表是合理的。

## 9. 当前回声算法的产品特征

如果用一句话总结当前算法：

它是一个“规则优先、轻量、可解释”的回声系统，而不是 AI 生成系统。

当前特征很明确：

- 可解释性强
- 行为稳定
- 成本低
- 易调试
- 个性化程度有限
- 没有真正的语义理解
- 没有情绪推理
- 没有基于训练内容深度生成文案

## 10. 当前实现的几个关键限制

### 10.1 回声只在新建记录时生成

这意味着：

- 如果用户后来修改训练部位、时长、状态
- 旧回声不会自动重算

### 10.2 未来话没有语义匹配

当前未来话复用只看：

- 是否最近刚用过
- 是否来自当前用户
- 是否不是当前记录自己写的

不看：

- 训练部位是否相关
- 训练状态是否相关
- 体重变化是否相关
- 文本语义是否相关

### 10.3 默认系统文案数量很少

当前只有 3 条。

所以如果用户没有写未来话，长期使用后回声的重复感会比较明显。

### 10.4 事实回声规则较简单

当前事实回声只覆盖两类信息：

- 近 7 天第一次练某个部位
- 近 7 天第 N 次训练

还没有覆盖：

- 时长变化
- 体重变化
- 连续训练天数
- 停训后回归
- 单部位训练频率变化

## 11. 推荐你怎么理解当前回声系统

如果后续你要继续迭代这个模块，建议把它拆成 3 层理解：

### 第 1 层：消息素材层

对应：

- `echo_message`

目前主要是用户自己写的未来话。

### 第 2 层：规则选择层

对应：

- `EchoService.chooseEcho(...)`

这里定义优先级和选择顺序，是当前算法核心。

### 第 3 层：结果快照层

对应：

- `echo_history`

这里记录最终发给某条训练记录的实际文案。

## 12. 一句话结论

当前回声算法是：

- 新建训练记录时触发
- 优先使用用户过去写下的未来话
- 其次用最近 7 天训练事实生成一条简短反馈
- 最后用默认系统文案兜底

它的优点是简单、稳定、容易解释；它的限制是个性化和智能程度还比较初级。
